# Velocity-Impulse Framework（VIF）設計書

## 1. 概要

本フレームワークは、「連続的なアウトプット生成」を**離散的なインパルス（単位行動）の反復**としてモデル化し、速度（Velocity）と質（Quality）のトレードオフを制御するための汎用モデルである。

対象領域：

* ライティング
* タスク処理
* コンテンツ生成
* ソフトウェア処理制御
* 認知負荷管理

---

## 2. 基本概念

### 2.1 インパルス（Impulse）

最小単位のアウトプット行動。

例：

* 1文を書く / 1コードブロックの実装
* 1つのアイデアの具体化
* 1件のリクエスト処理

---

### 2.2 ベロシティ（Velocity）

単位時間あたりのインパルス発生頻度。
離散値：0〜10

* **0：** 停止（Recoveryモード）。出力はゼロだが、疲労回復が最大化される。
* **1〜3：** 低速。深い思考、構造設計、精密な推敲。
* **4〜7：** 中速。バランスの取れた通常のアウトプット。
* **8〜10：** 極めて高速。質を度外視した発散・初稿生成。

---

### 2.3 強度（Strength）

1インパルスあたりに投入されるエネルギー量、または情報密度。内部ロジックでは「データ密度」として処理される。ベロシティが極大化する際、1インパルスあたりの強度は分散を抑えるために一定の閾値以下に制限される。

---

### 2.4 品質（Quality）

最終的なアウトプットが目標仕様や文脈に適合している度合い。
ベロシティおよび蓄積された疲労度（Fatigue）と負の相関関係を持つ。

---

### 2.5 出力（Output）

一定時間内に生成された総成果。

---

## 3. 数理モデル

### 3.1 インパルス生成と総出力

時間 $T$ における総出力は、各インパルスの強度 $f_i$ の総和として定義される。

$$Output(T) = \sum_{i=1}^{N(T)} f_i$$

---

### 3.2 ベロシティ、頻度、および慣性

ベロシティの変更には慣性（Inertia）が生じ、目標速度に達するまでの「ウォームアップ時間」が必要となる。速度が動的に変動するため、インパルス総数は以下の時間積分により算出される。

$$N(T) = \lfloor \int_{0}^{T} \lambda(v(t)) dt \rfloor$$

$$\lambda(v) = \lambda_{min} + (\lambda_{max} - \lambda_{min}) \times \frac{v-1}{9}$$

$$v(t) = v(t-1) + \alpha (v_{target} - v(t-1))$$

* $\alpha$：平滑化係数（$0 < \alpha \leq 1$）。熟練度 $S$ に比例し、目標速度への収束性を決定する。これにより急激な速度変化による数値的な発散を防止する。

---

### 3.3 疲労、回復、およびフロー

疲労度 $F(T)$ は、時間経過に伴う実効ベロシティと投入強度の積の蓄積として定義される。

$$F(T) = \int_{0}^{T} v(t) \cdot f(t) dt$$

インパルス強度は累積回数 $i$ と時間経過により減衰するが、$v=0$ において回復関数 $R$ が機能し、実効的な累積負荷を軽減する。特定の条件下（高 $Quality$ かつ高 $v$）では、フロー補正により疲労係数が抑制される。

$$f_i = f_{base} \times \exp(-k_{eff} \times (i - R(t_{rest})))$$

* $k_{eff} = k \times \phi$ （$\phi < 1$：フロー状態での疲労抑制係数）
* $R(t_{rest})$：$v=0$ 時の経過時間に基づく負荷リセット関数。

---

### 3.4 品質、熟練度、および認知オーバーフロー

品質はベロシティ $v$ に反比例し、ユーザーの熟練度 $S$ によって減衰が抑制される。また、ベロシティと疲労 $F$ の積が限界値を超えた場合、品質は強制的にゼロ（破綻）となる。

$$Quality(v, F) = \frac{1}{1 + (\frac{\beta}{S})v + \gamma F}$$

$$If (v \cdot F) > Threshold \implies Quality = 0$$

---

### 3.5 割込みコスト（Interruption Cost）

外部要因による強制停止後の再開時には、コンテキスト・スイッチによる出力減衰が発生する。

$$f_{start} = f_{base} \times (1 - \delta)$$

* $\delta$：中断時間とタスク複雑性に比例する減衰係数。

---

## 4. 制約・安定化モデル

### 4.1 キャパシティ制限

$$\lambda(v) \leq \lambda_{cap}$$

---

### 4.2 状態管理：Zero-Lock（ABCモデル）

システムの安定性を担保するため、バックエンドでは以下の動的なABCモデルによる状態保持を行い、ロジックのクラッシュを防止する。高ベロシティ時には、Boundaryの計算負荷を最適化（スロットリング）し、リアルタイム性を維持する。

* **Action（行動）：** 現在の実効速度および入力中のインパルス・ジャーナル。
* **Boundary（境界）：** 現在の疲労度から逆算される動的な「安全速度域」。リソース限界時には低精度の近似計算へ切り替える。
* **Context（文脈）：** 累積疲労、セッション履歴、および中断フラグ。

---

### 4.3 破綻とソフトリカバリ（Breakdown & Soft Recovery Lock）

疲労が閾値に達した際、または認知オーバーフローが発生した際、システムは「クールダウン」状態へ移行する。
この状態では、入力を完全に破棄するのではなく、裏側のバッファにデータを保持しつつ画面上は休息を促す演出を表示する。再始動アクション（画面タップ等）が行われた際、中断直前のUIパラメータ（スライダー位置等）を完全に復元してセッションを再開する。

---

## 6. 制御戦略

### 6.1 最適化関数（正規化）

$$Maximize: w_1 \cdot (Output \times Quality) - w_2 \cdot Cost(Fatigue)$$

---

### 6.2 最適化ルート（経路計画）

* **集中ルート：** 低速から徐々に上げ、ピーク後に $v=0$ で回復。
* **ポモドーロルート：** 一定の $v=6$ と短時間の $v=0$ を反復。
* **バーストルート：** 短時間の $v=10$ に特化し、即座に整理へ移行。

---

### 6.3 暗黙的フィードバック・ループ

BackSpaceキーの頻度や、推敲時のテキスト削除率をバックグラウンドで計測し、熟練度 $S$ や疲労係数 $k$ を自動調整する。ユーザーに自己評価を強制せず、行動データからモデルの適合精度を高める。

---

## 7. プロダクト実装指針

### 7.1 UI/UX要素（スマートフォン最適化）

* **ユニバーサルスライダー：** 0〜10のベロシティを片手操作可能な大型スライダーで制御。
* **状況の視覚化（Color Temperature）：** 現在の品質状態を直感的に伝えるため、UIの色温度を変化（青＝高品質、赤＝限界接近）させる。
* **触覚フィードバック（Haptic）：** キャパシティ接近時の重み付けや、ロック状態での拒絶振動。
* **状況ゲージ：** 「Flow Zone」と「Fatigue」を視覚的に統合表示。

---

### 7.2 状態保持・永続化アーキテクチャ

* **View/Modelの分離：** 状態異常時もユーザーが入力中のデータは独立して保持する。
* **インパルス・ジャーナリング：** 各インパルスを確定ごとにローカルDBへ即時書き込み、クラッシュ後の完全復元を実現する。
* **Zero-Lockマネージャー：** 操作系と数理モデル間の状態矛盾をロックフリーに同期する。

---

### 7.3 バックエンド解析（完全エッジ実行）

* **ローカルNLPエンジン：** 外部API依存を排除し、ネットワーク遅延や認証エラーによる中断を回避するため、軽量なPython NLPライブラリ等を端末内で直接実行し、リアルタイムに文脈・感情解析を行う。

## 8. 技術実装リファレンス

### 8.1 状態管理ロジック例（Python）

```python
class VIFController:
    def __init__(self, s_factor):
        self.v = 0.0
        self.f = 0.0
        self.alpha = 0.1 * s_factor # 熟練度に応じた追従性
        self.journal = []

    def update_velocity(self, v_target):
        # 目標速度への滑らかな遷移
        self.v += self.alpha * (v_target - self.v)
        
    def process_impulse(self, strength):
        if self.v * self.f > THRESHOLD:
            return "COOL_DOWN_REQUIRED"
        # 疲労蓄積の計算
        self.f += self.v * strength
        self.journal.append(strength)
        return "SUCCESS"
```

### 8.2 状態遷移の抽象化（Haskell概念）

```haskell
-- 状態モナドを用いたインパルス処理の抽象化
type ImpulseState = State VIFContext

applyImpulse :: Impulse -> ImpulseState Quality
applyImpulse impulse = do
    ctx <- get
    let newF = fatigue ctx + (velocity ctx * strength impulse)
    let newQ = calculateQuality (velocity ctx) newF (skill ctx)
    put ctx { fatigue = newF }
    return newQ
```

## 10. 本フレームワークの価値

* **抽象的勢いの定量化：** 「調子」を数理モデルとして扱える。
